== Testkonzept

=== Testobjekte und Teststufen Die Tests orientieren sich an der klassischen Testpyramide und decken mehrere Teststufen ab:

* Modultests:
API-Endpunkte und Funktionen mit gemockter Datenbank

* Integrationstests:
Zusammensoiel von API, Routing, HTTP-Statuscodes mit gemockter Datenbank

* Systemtests:
Zusammenspiel von Frontend und Backend aus Anwendersicht, manuelles Testen

=== Testmethoden

==== Backend Für das Backend werden automatisierte Tests eingesetzt:

pytest als Testframework

unittest.mock (MagicMock, patch) zur Simulation der Datenbankschicht

FastAPI TestClient zur Simulation von HTTP-Anfragen

Swagger UI zur manuellen Überprüfung und Exploration der API-Endpunkte

Die Datenbank wird in den Tests vollständig gemockt, sodass keine echte Verbindung zu einer produktiven oder lokalen Datenbank notwendig ist. Dadurch bleiben die Tests isoliert, reproduzierbar und unabhängig von der Laufzeitumgebung (z. B. VM).

==== Frontend Das Frontend wird durch manuelle Tests überprüft:

Starten der Anwendung im Browser

Durchklicken der Benutzeroberfläche

Überprüfung der in dem Pull-Request neuen Funktionalität

Dadurch Testen, wie ein Nutzer die Oberfläche sieht und bedient, wie Fehlermeldungen wirken und ob sowhol Design als auch Funktionalität aus Nutzersicht passen.

== Testergebnisse

=== Ergebnisse der Testdurchführung

Die automatisierten Backend-Tests werden erfolgreich ausgeführt, sofern alle notwendigen Abhängigkeiten installiert sind und die Tests korrekt gemockt werden.

Zentrale API-Endpunkte (z. B. Login, Root-Endpunkt) liefern die erwarteten HTTP-Statuscodes und Antwortdaten.

Die manuell getesteten Frontend-Funktionen verhalten sich erwartungsgemäß und ermöglichen eine fehlerfreie Bedienung der Anwendung.

Während der Testdurchführung wurde festgestellt, dass das Backend beim Import der API-Module zwingend gesetzte Umgebungsvariablen erwartet.

Dies führt lokal zu Fehlern (KeyError), wenn die Anwendung außerhalb der VM getestet wird.

Konsequenzen:

Für Tests werden Dummy-Umgebungsvariablen verwendet oder über Fixtures gesetzt.

Die Problematik ist dokumentiert und kann zukünftig durch eine robustere Konfigurationslogik (z. B. Default-Werte oder Konfigurationsdateien) verbessert werden.

Bekannte, noch nicht behobene Bugs werden in den GitHub Issues dokumentiert und mit dem Label bug gekennzeichnet.
